## State Layers

### Layer 1: Infrastructure (WebRTC + iOS Audio)
- **WebRTC ICE Connection**: Network connectivity state
- **RTCAudioSession**: iOS audio hardware I/O state

### Layer 2: Protocol (OpenAI Realtime API)
- **Response State Machine**: Tracks if AI is generating a response
- **Assistant Audio Streaming**: Tracks if audio data is being sent

### Layer 3: Detection (Speaking Detection)
- **RTP-Based Detection**: Polls audio levels from RTP stats
- **Event-Based Detection**: Uses OpenAI events as signals

### Layer 4: Application (Local Audio)
- **AudioMixPlayer**: Plays local sound effects (beeps, etc.)

### State Descriptions

| State | Description | Audio Monitoring |
|-------|-------------|------------------|
| `new` | Initial state | Not started |
| `checking` | ICE negotiation in progress | Not started |
| `connected` | ICE connection established | **Starts inbound/outbound monitoring** |
| `completed` | ICE negotiation complete, all candidates gathered | **Monitoring active** |
| `failed` | ICE connection failed | **Stops monitoring** |
| `disconnected` | Temporarily lost connection | **Stops monitoring** |
| `closed` | Connection closed | **Stops monitoring** |

## Known Issues & Edge Cases

### 1. RTP Detection Delay

**Issue**: `response.audio.done` fires immediately when server stops, but RTP-based `ğŸ”ŠğŸ”‡` detection can take up to 1 second longer due to `silenceRequiredDuration`.

**Impact**: UI may show "speaking" state after audio has stopped.

**Mitigation**: Use `assistantAudioStreaming` (event-based) for UI, RTP for backup/validation.

### 2. Audio Overlap Protection Gaps

**Issue**: If `assistantAudioStreaming` is set to `false` too early (e.g., `response.audio.done` received but playback buffer not yet drained), AudioMixPlayer may play over remaining audio.

**Current Solution**: Block playback while `assistantAudioStreaming = true`.

**Better Solution**: Combine with RTP silence detection before allowing local playback.

### 3. Queued Response TTL Expiration

**Issue**: If tool execution takes > 30 seconds, queued `response.create` expires.

**Impact**: AI won't respond to tool result.

**Mitigation**: Log expiration warning; consider increasing TTL or retrying.

### 4. Race Conditions in Response State

**Issue**: Multiple threads can check/modify `responseInProgress`.

**Current Solution**: All access through `responseStateQueue` (serial DispatchQueue).

### 5. WebRTC Audio Session Lifecycle

**Issue**: `audioSessionDidStopPlayOrRecord` fires when entire WebRTC session stops, not per-utterance.

**Impact**: Cannot use for per-response playback completion detection.

**Solution**: Use RTP stats polling for ongoing audio level detection.

## Recommendations for State Machine Rewrite

1. **Unified State Object**: Create a single `AudioSessionState` struct/class that holds all related states:
   ```swift
   struct AudioSessionState {
       var connectionState: RTCIceConnectionState
       var responseInProgress: Bool
       var assistantAudioStreaming: Bool
       var rtpSpeakingDetected: Bool
       var localPlaybackActive: Bool
       var pendingToolCalls: [String: ToolCallState]
   }
   ```

2. **State Machine Protocol**: Define clear state transitions:
   ```swift
   protocol AudioStateMachine {
       func transition(event: AudioEvent) -> AudioSessionState
   }
   ```

3. **Event Consolidation**: Map all events to a unified enum:
   ```swift
   enum AudioEvent {
       case openAIEvent(type: String, payload: [String: Any])
       case rtpStatsUpdate(speaking: Bool, level: Double)
       case audioSessionEvent(started: Bool)
       case toolCallEvent(callId: String, phase: ToolCallPhase)
   }
   ```

4. **Derived States**: Compute UI-relevant states from the core state:
   ```swift
   var isAssistantSpeaking: Bool {
       assistantAudioStreaming || rtpSpeakingDetected
   }
   
   var canPlayLocalAudio: Bool {
       !assistantAudioStreaming && !rtpSpeakingDetected
   }
   ```

5. **Logging Integration**: Emit state transitions as structured logs for debugging.
